Chapter 1

System Concept & End-to-End Workflow

Friday AI is an advanced, real-time predictive trading assistant designed to democratize high-frequency technical analysis and programmatic risk management. Positioned as an intelligent agent operating directly in the workspace of retail investors, Friday AI consolidates market scanning, technical charting, algorithmic trade execution, dynamic risk thresholds, and NLP-driven market explanations into a singular, highly cohesive platform.

System Architecture Workflow

The system operates through an asynchronous, distributed execution pipeline. Raw candlestick coordinates are fetched from exchange systems, parsed, and evaluated on the fly. The workflow operates as follows:

💡 Core Architecture Pillars
  • Asynchronous Thread Isolation: The FastAPI server manages the REST endpoints concurrently while spawning isolated WebSocket communication rings and safety check routines that prevent container lag.
  • Offline Machine Learning Execution: Model binaries are trained and compiled in local developer environments using historic multi-decade indices to control space complexities, and loaded instantly on startup as pre-trained modules.

Frameworks & Libraries Stack

To implement this system, several modern frameworks and specialized libraries were chosen to guarantee real-time latency and performance:

  • FastAPI (Python): Serves as the high-throughput web server gateway. Chosen for its native support for async/await execution, automated Pydantic request validation, and built-in WebSocket support.
  • Scikit-Learn & Joblib: Standardized data science utilities. Scikit-learn trains the Random Forest classifier model. Joblib packages and decompresses the trained binary into memory instantly.
  • yFinance (Yahoo Finance): Pulls real-time intraday candlestick frames from international exchanges.
  • Lightweight Charts (TradingView): The visual client library that compiles canvas layers dynamically, allowing users to draw overlays and trace price coordinate feeds in real-time.
  • Three.js & GSAP: Implements Awwwards-worthy immersive design. Three.js renders a 3D animated particle background, while GSAP controls the spring elastic cursor trailing and dock entrance sequences.
Chapter 2

Backend Architecture & Core Execution Logic

The backend of Friday AI is engineered using Python 3.10 and FastAPI. It operates as a stateless RESTful service combined with a stateful real-time WebSockets connection manager to notify client browsers of price movements and stop-loss triggers.

Machine Learning Random Forest Engine

The core predictive engine uses a Random Forest Classifier model trained on historic daily candlestick bars. The model targets predicting whether the closing price in three periods (days) will be higher (UP) or lower (DOWN) than the current closing price.

Technical Indicator Feature Extraction

For each stock candlestick row in the historic database, the model extracts technical indicators as features:

  • RSI (Relative Strength Index): Measures price momentum over a 14-period window.
  • MACD & MACD Signal: Tracks moving average crossovers to log trend direction.
  • SMA 20 & SMA 50: Identifies support/resistance structures and baseline trend alignments.
  • Close-to-SMA Ratios: Calculates percentage distance from moving averages to avoid overbought peaks.
# Feature creation segment in backend training pipeline
import pandas as pd
import numpy as np

def extract_features(df):
    # Calculate SMA
    df['sma_20'] = df['Close'].rolling(window=20).mean()
    df['sma_50'] = df['Close'].rolling(window=50).mean()
    
    # Calculate RSI
    delta = df['Close'].diff()
    gain = (delta.where(delta > 0, 0)).rolling(window=14).mean()
    loss = (-delta.where(delta < 0, 0)).rolling(window=14).mean()
    rs = gain / loss
    df['rsi'] = 100 - (100 / (1 + rs))
    
    # Ratios
    df['ratio_sma20'] = df['Close'] / df['sma_20']
    df['ratio_sma50'] = df['Close'] / df['sma_50']
    
    # Target label: 1 if close in 3 days is higher, else 0
    df['target'] = (df['Close'].shift(-3) > df['Close']).astype(int)
    
    return df.dropna()

Real-Time Risk Management & Safety Thread

To protect capital, Friday AI runs a dedicated, background safety manager loop. When a user buys paper shares of a stock, the backend establishes a tracker inside a state dictionary. As new tick updates are pulled from yFinance, the safety checker triggers evaluations:

Stop Loss Triggering

If the last traded price falls below the calculated stop-loss threshold: \[P_{tick} \le P_{entry} \times (1 - \text{Stop Loss \%})\] The system triggers an automatic emergency exit order, updates the ledger, logs the event to the user's warning container, and issues a WebSocket alarm notification to play an acoustic alarm on the user's browser.

Drawdown Threshold Freezing

Every time the portfolio values update, the backend evaluates the aggregate portfolio drawdown from the starting balance: \[\text{Drawdown \%} = \frac{\text{Initial Balance} - \text{Current Value}}{\text{Initial Balance}} \times 100\] If this value exceeds the user's drawdown stop slider setting (adjustable from 2% to 15%), the system instantly flags trading_frozen = True. In this state, all future buy requests are blocked, open positions are liquidated, and a lockdown notification alerts the user.

Chapter 3

Frontend Interface & Interactive Canvas Engine

The frontend interface of Friday AI is implemented as a single-page application (SPA) using vanilla HTML5, CSS3, and JavaScript (ES6+). It interfaces with the backend via REST endpoints and WebSocket protocols.

Single Page Application (SPA) View Router

To keep the interface clean and organized, the dashboard wraps card sections into dedicated views (#view-home, #view-chart, #view-holdings, and #view-signals). When a tab is selected from the floating bottom navigation dock, the script updates class attributes:

  • Hides all sections using display: none class directives.
  • Appends the .active class to display the selected section (using display: block or display: grid).
  • Animates the card layout container entrance using a GSAP fade-in transition: 
    gsap.fromTo(targetView, { opacity: 0, y: 15 }, { opacity: 1, y: 0, duration: 0.4 });

TradingView Lightweight Charts Integration

The interactive charting interface uses the high-performance TradingView Lightweight Charts canvas. The canvas rendering lifecycle requires careful coordination:

  1. Invisible Containers: Lightweight Charts cannot render grid coordinates correctly if initialized inside a container with display: none (the container client width is computed as 0px, causing the canvas elements to crash or render with 0px width).
  2. Resize Trigger Solution: When switching views to the Chart tab, the Javascript router invokes a short delay (50ms) to let the DOM display the view wrapper, and then explicitly triggers: 
    chart.resize(container.clientWidth, container.clientHeight);
    This forces the grid coordinate columns to recalculate and adapt perfectly to the viewport width.
🎨 Awwwards Creative UI Interactivity
  • 3D Particle Space Network: Renders 200 drift particles inside a WebGL context using Three.js, adding subtle interactive parallax drift tied to the user's mouse movements.
  • GSAP Cursor Follower: A custom div tracks the mouse coordinates. An outer glowing follower div trails the cursor with an elastic delay, scaling up and changing colors when hovering over interactive buttons.
  • Apple-style Dock: The bottom menu capsule uses a dark semi-transparent glassmorphism style (`backdrop-filter: blur(25px) saturate(180%)`) with a high z-index, allowing background components to blur dynamically as they scroll behind it.
Chapter 4

Deployment & Pipelines Infrastructure

To achieve high uptime, Friday AI is hosted across a distributed hybrid cloud pipeline. The deployment separates the stateful processing server from the static browser frontend asset bundle.

Stateless Backend Hosting: Render

The FastAPI backend server is deployed on the Render Serverless Cloud. Because serverless containers on Render's free tier automatically sleep after 15 minutes of inactivity and are subject to tight memory limitations (512MB RAM), the backend architecture implements two key mitigations:

  • Offline Training: Model binaries are trained and compiled locally in developer environments using multi-decade indices, and the pre-trained `backend/model.joblib` binary is pushed to GitHub. This avoids high CPU/RAM usage during container spin-up, preventing out-of-memory crashes.
  • Keep-Alive Pings: The frontend scripts issue lightweight check pings on page load to awake the sleeping backend container, and WebSocket heartbeats maintain the active socket connection.

Static Frontend Hosting: GitHub Pages

The static frontend assets (`index.html`, `styles.css`, `app.js`, logos) are hosted on **GitHub Pages**. GitHub Pages operates as a static CDN delivery network, guaranteeing sub-second content delivery speeds globally. The static files communicate with the backend dynamically using HTTPS REST API requests and stateful WebSocket protocols.

Chapter 5

Technical Interview Q&A Companion

This companion guide compiles high-level and low-level technical interview questions and answers, designed to help developers explain the architecture, data science choices, and performance optimizations of the Friday AI platform during engineering interviews.

🗣️ Q1: Why did you choose a Random Forest Classifier instead of deep learning (e.g., LSTM) for price prediction?

Answer: LSTMs and neural networks are prone to severe overfitting on noisy financial timeseries data unless fed with millions of parameters, and they require massive GPU instances to run, which is impractical for free cloud tiers (like Render's 512MB RAM limit). Random Forests build an ensemble of decision trees, which reduces variance through bootstrap aggregation (bagging), handles multi-dimensional indicator features (RSI, SMA, MACD) without scaling issues, and executes predictions in constant time (\(O(T \times D)\)) with minimal memory usage.

🗣️ Q2: How did you handle the timezone shift between local users and the New York Stock Exchange?

Answer: Financial chart intervals are highly sensitive to exchange time alignment. If values are evaluated in local time (e.g., IST/GMT), the start and end of daily candles will shift, breaking candlestick calculations. To resolve this, the backend processes all historic stock queries in New York Exchange Time (America/New_York). On the client side, the Lightweight Charts library was configured with a custom local formatter: 

timeZone: 'America/New_York'
This ensures the tick marks on the X-axis align with the Wall Street session (EST/EDT) regardless of where the user is physically located.

🗣️ Q3: Explain the z-index and translation overrides you solved on the bottom navigation dock.

Answer: The floating navigation capsule is centered in CSS using left: 50% and transform: translateX(-50%). When animating the entrance using GSAP, assigning y: 50 caused GSAP to overwrite the element's inline transform styles to translate(0px, 50px), removing the translateX(-50%) offset and throwing the bar off-screen on mobile devices. I resolved this by migrating the animation to native CSS `@keyframes` using transform: translate(-50%, 50px) to translate(-50%, 0), which isolates the animation rendering from Javascript execution halts and guarantees visibility.

🗣️ Q4: How is the risk-management freeze thread designed to be thread-safe in FastAPI?

Answer: FastAPI operates on an asynchronous event loop. Instead of blocking OS threads which would choke API requests, the background portfolio monitoring loop uses Python's native asyncio task Scheduler. The safety checker runs await asyncio.sleep(2.5), returning control back to the event loop to process incoming trade requests, and uses async HTTP clients (like httpx or async routines) to update shared global portfolio records safely.

🗣️ Q5: What is the time complexity of the market scanning operation, and how is it optimized?

Answer: The time complexity of scanning \(M\) tickers with \(N\) candlesticks each is \(O(M \times N)\). Fetching data from external APIs sequentially would cause severe network bottlenecks. To optimize this, the backend implements multi-threaded caching. Daily scan profiles are parsed and cached in local memory, and yFinance queries pull batch indexes rather than single historical lines, reducing scan time from 30 seconds to under 2 seconds.

Evolution Timeline, Challenges & Roadmaps

This chapter documents the software engineering journey of Friday AI, detailing how the system evolved from a simple Python script to a fully interactive, production-grade cloud platform.

Evolution Timeline

  • Phase 1: CLI Prototype & Data Pipeline: Built the initial Python scripts to interface with yFinance, extract features (RSI, SMA), train the machine learning classifier, and output buy/sell signals in the console.
  • Phase 2: FastAPI API Gateway: Wrapped the CLI scripts into RESTful endpoints. Introduced sqlite databases to persist paper trading transactions and daily drawdown settings.
  • Phase 3: Interactive Dashboard UI: Built the HTML dashboard using the TradingView Lightweight Charts canvas. Added simple AJAX polling to update prices every 5 seconds.
  • Phase 4: Creative UI Redesign: Upgraded the user interface to implement Awwwards-worthy design rules. Integrated Three.js 3D background particle clouds, GSAP cursor tracking, and a glassmorphic bottom navigation dock.
  • Phase 5: Single Page Application (SPA) Migration: Split the dashboard into dedicated tab views (Portfolio, Chart, Holdings, Signals) to avoid vertical page clutter. Developed the native CSS spring-elastic slide entrance animations to guarantee mobile responsiveness.

Technical Challenges Faced & Solutions

🛠️ Challenge 1: Render Out-of-Memory Container Crashes

Details: Render's free hosting tier enforces a strict memory limit of 512MB RAM. Initially, when the FastAPI backend spun up, it executed the feature extraction and Random Forest model training pipeline on historical data in real-time. This process spiked memory consumption above 600MB, triggering immediate container kills by the hosting environment.

Solution: Moved the model training pipeline offline to the developer's local machine. The trained estimator is saved to a compressed binary file (model.joblib) and pushed to GitHub. The Render backend simply loads the pre-trained binary on startup, reducing memory usage from 600MB to under 150MB.

🛠️ Challenge 2: Mobile Viewport Dock Button Overflow

Details: The bottom floating nav dock was squishing on mobile screens under 400px wide. The "Chat Assistant" button had long text, which overflowed the nav bar boundaries, pushing elements off-screen and locking users out of the chat interface.

Solution: Added media query rules to hide the logo and dividers on screens under 600px wide, and collapsed the "Chat Assistant" button into a circular floating button containing only the robot icon. This saved over 180px of horizontal space, letting the entire navigation bar display beautifully on mobile viewports.

Future Implementation Roadmap

  1. Live Brokerage API Integration: Replace paper trading endpoints with secure REST Webhooks to connect with retail brokerage APIs (such as Interactive Brokers or Alpaca), enabling live trade execution directly from the Friday AI command center.
  2. Historical Portfolio Backtester: Build an offline backtesting engine that simulates the Random Forest model's performance over multi-decade cycles, generating equity curves and Sharpe ratio reports.
  3. Advanced NLP Conversational Memory: Integrate stateful chat histories using Redis cache databases to allow the Gemini LLM assistant to recall past user portfolio questions during ongoing conversations.
