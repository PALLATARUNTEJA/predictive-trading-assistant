Chapter 1

The Core System Concept & Flow Explained for Beginners

Welcome to the developer guide for **Friday AI**! If you are a 10th-grade student or a beginner who only knows basic coding syntax, this chapter is written specifically for you. We will use simple analogies to explain exactly what Friday AI does and how it functions under the hood.

1. The Restaurant Analogy of APIs

Imagine you walk into a restaurant. You are the **Client** (the browser window). The kitchen is the **Database & Python Code** (where the food is prepared). You cannot go directly into the kitchen to grab food. Instead, you look at a menu and tell the waiter what you want. The waiter is the **API (Application Programming Interface)**.

When you click "Portfolio" on Friday AI, the browser (client) asks the waiter (FastAPI) to go to the kitchen (database) and fetch your stats. The waiter returns with a plate of data formatted in **JSON** (a clean text structure that looks like a dictionary), and the browser displays it beautifully on your screen.

2. The Forecast Analogy of Machine Learning

How does Friday AI predict if a stock is going UP or DOWN? Think of it like a weather forecaster. If a forecaster wants to predict if it will rain tomorrow, they look at historical records: "Was it humid? Was the wind blowing from the East? Was the pressure dropping?"

Similarly, the **Random Forest Classifier** is our weather forecaster for stocks. Instead of humidity or wind, it looks at:

  • RSI (Relative Strength Index): Is the stock price rising too fast (hot/humid)?
  • SMA (Simple Moving Average): Is the price above its 20-day average line?
By checking historical patterns of 41,000 candles, it makes a statistical guess: "75% of the time when RSI was this high and SMA was crossed, the stock price went up over the next 3 days." This is machine learning in action!

3. WebSockets: The Continuous Walkie-Talkie

Normally, web communication is like sending letters: you send a request, you get a response, and the connection closes. But when a stock price is changing every second, sending letters is too slow.

We use **WebSockets** to establish a continuous "walkie-talkie" connection between the backend server and the frontend screen. Once you open the website, the connection stays open. If a stock drops below your stop-loss limit, the backend instantly shouts through the walkie-talkie: "ALARM! Stock sold!" and the browser plays the alarm sound immediately without you needing to refresh the page!

Section 3: WebSocket Connection Manager (Lines 26–46)

class ConnectionManager:
    def __init__(self):
        self.active_connections: List[WebSocket] = []

    async def connect(self, websocket: WebSocket):
        await websocket.accept()
        self.active_connections.append(websocket)

    def disconnect(self, websocket: WebSocket):
        self.active_connections.remove(websocket)

    async def broadcast(self, message: dict):
        for connection in self.active_connections:
            try:
                await connection.send_json(message)
            except Exception:
                pass

ws_manager = ConnectionManager()

This class manages active browser tabs connected to the server via WebSockets. The list self.active_connections stores all active users. When a user opens the page, connect() accepts the socket and adds it to the list. When they close the tab, disconnect() cleans it up. When a trade is made or stop-loss is triggered, broadcast() loops over every connected browser tab and sends them the transaction alarm JSON packet immediately.

Chapter 3

Deep-Dive into Core Logic & Algorithms

1. Random Forest Machine Learning Classification

A **Random Forest** is an ensemble classifier. Instead of using a single Decision Tree (which easily overfits and makes poor choices), it creates a "forest" of multiple independent trees (default is 100). Each tree is trained on a random subset of historical candlestick data and features.

When predicting direction:

  1. Each tree votes on whether the stock will go UP or DOWN.
  2. The system gathers all votes. If 75 trees vote UP, the prediction is UP with a confidence of 75%.
This bagging technique ensures the model is robust against stock price volatility.

2. Technical Indicator Mathematics

Relative Strength Index (RSI)

RSI measures price speed and change momentum over a 14-day window: \[RSI = 100 - \left( \frac{100}{1 + RS} \right)\] Where \(RS\) is the ratio of average upward price gains to average downward losses over the 14-period window: \[RS = \frac{\text{Average Gain}}{\text{Average Loss}}\] RSI values above **70** indicate the stock is overbought (too hot), while values below **30** indicate it is oversold (cheap/undervalued).

Moving Average Convergence Divergence (MACD)

MACD tracks the relationship between two Exponential Moving Averages (EMA): \[\text{MACD Line} = \text{EMA}_{12}(\text{Close}) - \text{EMA}_{26}(\text{Close})\] \[\text{Signal Line} = \text{EMA}_{9}(\text{MACD Line})\] When the MACD line crosses above the Signal line, it indicates accelerating upward momentum (a bullish buy trigger). When it crosses below, it indicates downward momentum (a bearish exit trigger).

Libraries & Frameworks Reference Guides

This chapter serves as a reference guide explaining the role of each library implemented in Friday AI:

  • FastAPI: A modern, high-performance web framework for building APIs in Python. It handles our HTTP routes, JSON serialization, and WebSocket connections.
  • Uvicorn: An ASGI web server implementation for Python. It executes the FastAPI app, listening for web requests on port 8000.
  • Scikit-Learn: A comprehensive data science library. Used to train the Random Forest estimator and run predictions.
  • Joblib: Provides lightweight pipelining and serialization. Used to save our trained model binary locally and load it into memory on the Render server in under 0.1 seconds.
  • yFinance: A Python wrapper that pulls historical market data and real-time intraday ticks directly from the Yahoo Finance database.
  • Lightweight Charts: TradingView's open-source canvas library. Used to render smooth candlestick charts on the frontend.
  • Three.js: A lightweight 3D library used to render particle animations in a WebGL canvas.
  • GSAP (GreenSock Animation Platform): A high-performance animation library. Used to animate cursor trailing, tab view transitions, and dock slide entrances.

50+ Comprehensive Interview Questions & Answers

This chapter contains exactly 50 technical interview questions and answers, categorized into five key sections, designed to prepare developers for engineering interviews.

Section A: Python & FastAPI Framework

Q1: What is the difference between an ASGI server (like Uvicorn) and a WSGI server (like Gunicorn)?

Answer: WSGI (Web Server Gateway Interface) is designed for synchronous Python frameworks (like Django or Flask). It handles one request per thread sequentially. ASGI (Asynchronous Server Gateway Interface) is WSGI's successor. It supports async/await, allowing a single thread to manage thousands of concurrent connections. This is essential for long-lived protocols like WebSockets or SSE (Server-Sent Events) used in Friday AI.

Q2: How does FastAPI use Pydantic for request validation?

Answer: Pydantic handles type enforcement on request bodies. We define a class inheriting from BaseModel (like BuyRequest) with typed fields. When a POST request arrives, FastAPI compares the JSON payload against our schema. If types mismatch (e.g., text instead of float for amount), FastAPI returns a 422 Unprocessable Entity response before executing any controller code.

Q3: What is CORS, and how is it configured in Friday AI?

Answer: CORS (Cross-Origin Resource Sharing) is a browser security mechanism that blocks webpages from making requests to a different domain than the one that served it. Because our frontend is hosted on GitHub Pages and our backend is on Render, the browser would block the requests. We configure CORSMiddleware in FastAPI, setting allow_origins=["*"] to tell the browser that our API can be safely read by external domains.

Q4: Why use async/await keywords in FastAPI endpoint declarations?

Answer: Declaring an endpoint as async def tells FastAPI that the function contains non-blocking operations. When the code awaits an operation (like database reads or WebSocket broadcasts), the event loop releases the thread to process other incoming API requests, dramatically increasing throughput under high concurrency.

Q5: How does FastAPI generate auto-interactive documentation?

Answer: FastAPI parses route decorators and Pydantic schemas on load. It generates an OpenAPI JSON specification, which is served via Swagger UI at /docs and ReDoc at /redoc, providing developers with an interactive testing playground.

Q6: What is WebSocketDisconnect, and how is it handled in theConnectionManager?

Answer: WebSocketDisconnect is raised when a client closes their browser tab or loses connection. In app.py, we wrap the WebSocket read loop in a try/except WebSocketDisconnect block. The catch statement calls ws_manager.disconnect(websocket) to remove the dead socket from our active connections list, avoiding memory leaks.

Q7: What is the purpose of FastAPI Dependency Injection?

Answer: Dependency Injection allows us to share utilities (like database sessions, security tokens, or CORS settings) across multiple routes. By passing a dependency as a function argument using Depends(), FastAPI automatically handles its instantiation and lifecycle management.

Q8: Why does the backend chat endpoint use Header parameters?

Answer: Instead of writing secret API credentials into query parameters (where they would be logged in plain text by web servers), we declare x_gemini_key: Optional[str] = Header(None). FastAPI automatically extracts HTTP request headers starting with `X-Gemini-Key` and routes them safely to the handler function.

Q9: Explain the difference between Query parameters and Path parameters in FastAPI.

Answer: Path parameters are embedded parts of the URL structure (e.g., /api/chart/{ticker}), representing core resources. Query parameters are appended at the end after a question mark (e.g., /api/scan?tickers=AAPL), representing options or filters.

Q10: What is the role of StaticFiles in app.py?

Answer: StaticFiles allows FastAPI to host static directory structures (HTML, CSS, JS). On lines 465–466, we mount the `frontend` folder to the root URL (/), allowing Uvicorn to serve the user interface during local testing.

Section B: Machine Learning & Technical Indicators

Q11: Why is the Random Forest Classifier model trained offline?

Answer: Training models requires heavy computation. On Render's free tier, memory is capped at 512MB RAM. Running feature extraction and forest training on thousands of candles spikes memory above 600MB, triggering container kills. Training the model offline and saving the binary via Joblib lets the Render instance load the model instantly, using under 150MB of RAM.

Q12: What are features in our machine learning dataset?

Answer: Features are the inputs our model uses to make predictions. In Friday AI, these are technical indicators derived from historical price data: RSI, SMA ratios, MACD crossovers, and volatility metrics.

Q13: What is the target variable (label) in our dataset?

Answer: The target is a binary label: 1 (UP) if the close price in 3 periods is higher than the current price, else 0 (DOWN). The model learns to map input technical indicators to this future outcome.

Q14: Explain how RSI indicator calculations can suffer from look-ahead bias if not careful.

Answer: Look-ahead bias occurs when future data leaks into the past during model training. To prevent this, RSI is computed using rolling historical calculations: we only compute the average gain/loss over the preceding 14 candles, never including future price movements in the feature set.

Q15: How does the Random Forest handle collinearity between indicators (like SMA 20 and SMA 50)?

Answer: High correlation (collinearity) between features can confuse linear regression models. Random Forests are tree-based and split nodes based on information gain, making them naturally robust against collinearity. The algorithm still selects the feature that provides the cleanest separation of labels at each node.

Q16: What is model serialization, and how is it used in Friday AI?

Answer: Model serialization is the process of converting an active Python ML model object in RAM into a static byte-stream file. We use Joblib to serialize the Random Forest model as model.joblib, which can be easily version-controlled and loaded on the server.

Q17: What does the model confidence score represent?

Answer: The confidence score is the percentage of trees in the forest that voted for the majority prediction. If 75 trees vote `UP` and 25 vote `DOWN`, the confidence is 75%. This is calculated using model.predict_proba().

Q18: Why is feature scaling (like normalization or standardization) unnecessary for Random Forest?

Answer: Random Forests make predictions by splitting data at vertical boundaries (e.g., RSI > 70). Because tree divisions are threshold-based, they are scale-invariant, meaning features do not need to be scaled to a common range.

Q19: Explain the risk of overfitting in Random Forests. How is it controlled?

Answer: Overfitting occurs when trees grow too deep and memorize noise. We control this by setting max depth limits (e.g., max_depth=12) and limiting the minimum number of samples required to split a node, forcing generalization.

Q20: What is out-of-bag (OOB) error in Random Forest?

Answer: OOB error is a built-in cross-validation metric. Because each tree is trained on a random subset of data (bootstrap sample), we can evaluate its accuracy using the remaining data it hasn't seen, providing an unbiased validation metric.

Section C: Frontend Design & UI Interactivity

Q21: Why does TradingView's Lightweight Charts require a resize trigger on tab switches?

Answer: When a tab is hidden with display: none, its parent dimensions collapse to 0px. If Lightweight Charts is initialized inside it, the canvas width is calculated as 0px, causing it to crash or render with 0px width. When switching tabs, we use a 50ms delay to let the DOM display the view, then call chart.resize() to force grid recalculations.

Q22: How is the iOS-style spring animation implemented for the floating bottom navigation dock?

Answer: It is implemented in CSS using native keyframe animations: 

animation: slideUpDock 0.7s cubic-bezier(0.34, 1.56, 0.64, 1) 0.8s forwards;
The cubic-bezier values create a spring-like ease that bounces slightly past the target position before settling, matching standard iOS animations.

Q23: How does the GSAP custom cursor follower track the mouse?

Answer: We bind a mousemove listener to the window: 

window.addEventListener("mousemove", (e) => {
    gsap.to(cursor, { x: e.clientX, y: e.clientY, duration: 0.05 });
    gsap.to(follower, { x: e.clientX, y: e.clientY, duration: 0.22, ease: "power2.out" });
});
The longer duration on the follower element creates a smooth, elastic lag behind the actual cursor position.

Q24: What is the purpose of backdrop-filter in the bottom navigation dock CSS?

Answer: backdrop-filter: blur(25px) saturate(180%) applies visual filters to the elements *behind* the semi-transparent navigation dock, creating a modern glassmorphic blur effect where text and images blur dynamically as they scroll behind the bar.

Q25: How is the WebGL particle network initialized in Three.js?

Answer: We create a BufferGeometry and populate it with 200 random 3D coordinates. We then apply a PointsMaterial with a custom color and bind it to a Points mesh, rendering it within a WebGL context container.

Q26: How does the Three.js background achieve interactive parallax drift?

Answer: We track the normalized mouse positions in a global variable. In our WebGL render loop, we update the particle system's rotation coordinates: 

particles.rotation.y += (targetX - particles.rotation.y) * 0.05;
The 0.05 interpolation factor creates a smooth, lagging parallax drift as the user moves their mouse.

Q27: What is the z-index property, and how is it used in Friday AI?

Answer: `z-index` controls the stacking order of elements on a page. The 3D canvas is placed at the back (z-index: -2), while critical elements like the bottom nav dock (z-index: 9999999) and modals are placed at the front to remain visible.

Q28: How does the SPA view toggler work in vanilla JS?

Answer: Clicking a tab trigger button fires an event listener that loops through all `.dashboard-view` elements, removing their `.active` class to hide them, and adds the `.active` class to the targeted container ID to display it.

Q29: Explain the purpose of scale transformation animations on button active clicks.

Answer: Applying transform: scale(0.95) on active clicks provides immediate tactile feedback, making the interface feel responsive and alive.

Q30: Why was the y-axis translation removed from the GSAP dock animation in favor of CSS?

Answer: The nav dock is centered horizontally using left: 50% and transform: translateX(-50%). Animating `y` in GSAP overwrote the element's transform property to `translate(0px, y)`, removing the horizontal centering and throwing the bar off-screen. We resolved this by migrating the animation to native CSS `@keyframes` which isolates the animation rendering from Javascript execution halts.

Section D: System Design & WebSockets

Q31: What is a WebSocket handshake?

Answer: A WebSocket connection begins with a standard HTTP GET request containing an `Upgrade: websocket` header. If the server supports the protocol, it returns an `HTTP 101 Switching Protocols` status code, upgrading the connection to a persistent, bi-directional TCP socket stream.

Q32: Why are WebSockets preferred over AJAX polling for stop-loss alerts?

Answer: AJAX polling requires the browser to send HTTP requests to the server every few seconds (e.g. 5s) to check for updates. This consumes high bandwidth, increases database load, and introduces a delay of up to 5 seconds. WebSockets keep a persistent TCP channel open, allowing the server to push stop-loss alerts to the browser instantly the moment they are triggered.

Q33: How does the ConnectionManager broadcast messages to users?

Answer: It loops through all active WebSocket connection objects stored in the `self.active_connections` list and calls: 

await connection.send_json(message)
We wrap each send call in a try/except block to handle and discard dead connection handles silently.

Q34: How is drawdown calculated in the backend safety manager?

Answer: Drawdown is calculated by comparing the portfolio's current total value (NAV) to the starting balance ($10,000.00): \[\text{Drawdown \%} = \frac{10000 - \text{Total Value}}{10000} \times 100\] If this value exceeds the user's limit, the system locks trading to protect capital.

Q35: How does the backend prevent duplicate purchases if the user clicks a buy button multiple times?

Answer: We implement an event lock state inside the trading engine. When a transaction starts, the system checks the lock. If active, subsequent requests are blocked until the first transaction completes, avoiding double-execution bugs.

Q36: Explain the role of JSON in the REST API.

Answer: JSON (JavaScript Object Notation) is a lightweight, human-readable data format. It acts as the standard data transport container for our REST API endpoints, allowing the Python backend and JavaScript frontend to serialize and parse data tables effortlessly.

Q37: Why does the system use a stateless REST design for trades but stateful for WebSockets?

Answer: Stateless REST endpoints are easy to scale and cache because each request contains all the information needed to complete it (e.g. symbol, buy amount). WebSockets require stateful connections because the server must maintain active socket handles in memory to push real-time alerts to the browser.

Q38: How does the backend chat router fallback to local analysis under network drops?

Answer: If API requests to external LLM servers fail or time out, the Python backend catches the exception and routes the query to local indicator analyses (fetching predictions from SQLite and indicators from `app.py`), returning structured replies locally.

Q39: What is a Pydantic schema validation error status code?

Answer: A schema validation failure returns an `HTTP 422 Unprocessable Entity` status code, containing a JSON payload detailing the invalid variables or types.

Q40: How is the chatbot Gemini API call authenticated?

Answer: It reads the API key from the user-provided HTTP header X-Gemini-Key. The backend passes this key as a query parameter (?key=API_KEY) to Google's Generative Language API endpoint, keeping the key secure.

Section E: Hosting & Web Performance Optimization

Q41: What is the 512MB RAM limitation on Render's free tier, and how was it bypassed?

Answer: Render's free tier enforces a strict memory limit of 512MB. Running feature calculations and model training on startup exceeded 600MB, triggering container kills. We resolved this by training the model offline and saving the compiled binary as model.joblib. The Render instance simply loads this pre-trained binary on startup, reducing memory usage to under 150MB.

Q42: What is the cold-start delay on Render free instances, and how do we handle it?

Answer: Render free containers automatically sleep after 15 minutes of inactivity. When a new user visits the site, spinning up a sleeping container introduces a delay of 30–50 seconds. We handle this by sending a lightweight check request to the API on page load to wake up the server while the user interacts with the loading screen.

Q43: What is the time complexity of yFinance candlestick downloads?

Answer: Downloading \(N\) daily candles is \(O(1)\) relative to dataset size since it fetches pre-computed data tables from Yahoo Finance's global CDN databases, taking under 0.5 seconds.

Q44: What is the space complexity of our Random Forest model in memory?

Answer: The space complexity is \(O(T \times S)\), where \(T\) is the number of trees (100) and \(S\) is the average number of nodes per tree. For our pre-trained model, the file size is under 2.5MB, taking minimal space in memory.

Q45: How does hosting frontend assets on GitHub Pages improve load speeds?

Answer: GitHub Pages operates as a static CDN delivery network. It caches and serves static files (`index.html`, `styles.css`, `app.js`) from server edge locations closest to the user, providing sub-second load times globally.

Q46: How do we optimize network payload sizes between the API and browser?

Answer: We serialize dataset values to raw arrays: 

[{"time": "2026-05-01", "close": 315.4}, ...]
This structured array mapping removes text keys from rows, reducing packet sizes by up to 60%.

Q47: Why does the system use relative script imports on GitHub Pages?

Answer: Absolute root references (like /app.js) look for files at the domain root (e.g. pallatarunteja.github.io/app.js), which fails because the site is hosted in a subdirectory (/predictive-trading-assistant/). Using relative paths (app.js) ensures assets load correctly.

Q48: Explain the difference between HTTP and WebSockets resource utilization.

Answer: HTTP requests require opening a new connection, sending headers, and closing the socket for each request, consuming high CPU. WebSockets establish a single TCP connection, sending lightweight frames with minimal header overhead, reducing server resource utilization.

Q49: How is the database state persisted across Render container restarts?

Answer: SQLite databases inside Render containers are ephemeral and reset on restart. To persist portfolio data, the backend uses persistent disk volumes mounted at `/data/`, ensuring records survive server restarts.

Q50: Explain how Gzip compression optimizes Friday AI file delivery.

Answer: Gzip compresses text-based asset files (like `styles.css` and `app.js`) on the server before transmitting them over the network. The browser decompresses them automatically, reducing file transfer sizes by up to 75% for faster page loads.

Major Challenges & Resolutions

🛠️ Challenge 1: Render Out-of-Memory Container Crashes

Details: Render's free tier limits memory to 512MB RAM. Initially, the FastAPI backend ran data loading, indicator calculations, and model training in real-time on startup. This spiked memory consumption above 600MB, triggering container kills.

Solution: Moved the model training pipeline offline to the developer's local machine. The trained model is serialized to a compressed binary (model.joblib) and pushed to GitHub. The Render backend simply loads this pre-trained binary on startup, reducing memory usage to under 150MB.

🛠️ Challenge 2: Mobile Viewport Dock Button Overflow

Details: On mobile screens under 400px wide, the floating bottom navigation dock squished and overflowed. The "Chat Assistant" button had long text, which pushed elements off-screen and locked users out of the chat interface.

Solution: Added media query rules to hide the logo and dividers on screens under 600px wide, and collapsed the "Chat Assistant" button into a circular floating button containing only the robot icon. This saved over 180px of horizontal space, letting the entire navigation bar display beautifully on mobile viewports.
